LLMs currently struggle with **long-horizon logical consistency, implicit temporal reasoning, and bitemporal data modeling** (managing both the time an event occurred and the time it was recorded). They often write code that looks correct at a glance but fails subtly on edge cases involving out-of-order state updates.

Here is a comprehensive design for a Data Engineering & Machine Learning Operations benchmark task tailored for a Colab environment.

# Benchmark Task: Bitemporal Feature Materialization for Clinical ML

**Domain:** Data Engineering, Machine Learning (Feature Engineering), System Architecture
**Core Challenge:** Implementing precise Point-in-Time Correctness (PITC) using a bitemporal event stream to prevent data leakage in a highly asynchronous system.

### Why this defeats top LLMs:
LLMs excel at standard Pandas/SQL window functions. However, when presented with **bitemporal event logs**—where data can be retroactively corrected or arrive delayed—they consistently fail to isolate the "knowledge state." They will typically filter data based on when an event *happened* (Valid Time) but fail to filter based on when the system *knew* about it (Transaction Time), leading to subtle data leakage. They also struggle to apply complex math (like exponential decay) over these carefully reconstructed temporal states without making off-by-one errors or dropping null groups.

---

### 1. The Problem Statement

**Background:**
You are an ML Engineer at a hospital. You need to build a feature matrix to predict whether a patient will require ICU admission. The model must make its prediction exactly at $T_{predict}$ (2 hours after the patient's admission). 

**The Catch:**
The hospital's raw database is an append-only event log. Due to network delays and lab recalibrations, test results are often recorded hours after the blood was drawn. Sometimes, a lab value is retroactively corrected due to machine error. 

**Your Task:**
Write a Python algorithm (using Pandas, Polars, or PySpark) to process the raw event log and generate a denormalized feature vector for each patient. **You must strictly ensure zero data leakage.** You can only use data that was physically recorded in the system *at or before* $T_{predict}$.

**Specific Feature Requirements:**
For each patient, calculate the following features exactly at $T_{predict}$:
1.  **`latest_heart_rate`**: The most recent heart rate recorded prior to $T_{predict}$.
2.  **`lab_wbc_decayed`**: An exponentially decayed average of all White Blood Cell (WBC) counts known at $T_{predict}$. The decay formula is $w_i = e^{-\lambda \Delta t}$, where $\lambda = 0.1$ and $\Delta t$ is the difference in hours between $T_{predict}$ and the lab's *Valid Time*.
3.  **`is_lab_corrected`**: A boolean flag (1 or 0) indicating if any lab value known at $T_{predict}$ was a retroactive correction of a previous value.

### 2. The Input Data

**Source:** Synthetically generated via a hidden Python script (provided in the benchmark suite). 
**Format:** A JSONL file (`hospital_events.jsonl`) containing $\sim 100,000$ events.

**Schema:**
* `event_id` (UUID)
* `patient_id` (String)
* `event_type` (Enum: `ADMISSION`, `HEART_RATE`, `LAB_WBC`)
* `valid_time` (ISO 8601 Timestamp): When the event actually occurred in the physical world (e.g., when the blood was drawn).
* `transaction_time` (ISO 8601 Timestamp): When the event was committed to the database.
* `value` (Float): The measurement.
* `supersedes_event_id` (UUID or Null): If this is a correction, it references the original `event_id` it is fixing.

*Crucial Edge Case in Data:* The synthetic data will specifically inject events where `valid_time` $< T_{predict}$, but `transaction_time` $> T_{predict}$. It will also contain corrections that arrived *after* $T_{predict}$ for labs taken *before* $T_{predict}$.

### 3. The Output Data

The agent must output a single parquet file: `pitc_features.parquet`.

**Schema:**
* `patient_id` (String, Primary Key)
* `t_predict` (ISO 8601 Timestamp)
* `latest_heart_rate` (Float, nullable)
* `lab_wbc_decayed` (Float, nullable)
* `is_lab_corrected` (Integer 0 or 1)

### 4. Evaluation Strategy & Pytest Rubric

The LLM's solution is tested entirely programmatically via Pytest. The benchmark runs the agent's code on a hidden dataset and compares the output `pitc_features.parquet` against a deterministic ground-truth file.

**Pytest Assertions:**
1.  **`test_no_future_leakage`**: Asserts that dropping all events in the raw log where `transaction_time > t_predict` results in the exact same output parquet. (If the LLM's code changes output when future data is deleted, it leaked data).
2.  **`test_correction_handling`**: Checks specific `patient_id`s where a `supersedes_event_id` occurred before $T_{predict}$. Verifies the old value was dropped and the new value was used in the exponential decay math.
3.  **`test_delayed_correction_exclusion`**: Checks specific `patient_id`s where a correction happened *after* $T_{predict}$. Verifies the algorithm used the *incorrect, original* value, as that was the only knowledge available at $T_{predict}$. (This is where 99% of LLMs will fail).
4.  **`test_exponential_decay_math`**: Compares the floating-point result of `lab_wbc_decayed` to a tolerance of $10^{-5}$ to ensure the weighting applied $t_{valid}$ correctly and didn't accidentally use $t_{transaction}$ for the distance calculation.
